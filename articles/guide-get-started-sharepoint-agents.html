Get Started with Agents in SharePoint and Custom Copilot Agents

A clear, practical guide to the no-code SharePoint agent experience, advanced Copilot Studio agents, licensing, limits, governance, and the PowerShell checks admins should run before rollout.

Copilot Updated May 29, 2026 18 min read

SharePoint agents are one of the most practical ways to bring Microsoft 365 Copilot closer to the work people already do. Instead of asking employees to search through libraries, policy pages, and project folders, an agent lets them ask plain-English questions and get answers grounded in the SharePoint content they already have permission to see.

Short version: Use a built-in SharePoint agent when you want a fast, no-code Q&A assistant over a site, library, folder, or selected files. Use a Copilot Studio agent when you need actions, workflows, external systems, advanced topics, or a more controlled enterprise rollout.

Quick Answer: Which Agent Should You Build?

Option 1: SharePoint built-in agent

Best for site owners who want a no-code assistant that answers questions from SharePoint pages, documents, libraries, folders, or selected files.

Option 2: Copilot Studio agent

Best for IT, HR, operations, and service teams that need multi-step conversations, Power Automate actions, external data, or controlled deployment.

License path

Users need Microsoft 365 Copilot, or the organization must enable pay-as-you-go/capacity for SharePoint agents where supported.

Security model

Agents respect SharePoint permissions. If a user cannot access a source, the agent should not use that source in its answer for that user.

What Microsoft's Official Guidance Says

Microsoft describes SharePoint agents as AI assistants for questions about sites, pages, libraries, lists, and files the asker already has permission to access. That last part matters: agents sit on top of SharePoint permissions; they do not replace them.

  • Every SharePoint site gets a ready-made agent: It appears by default, is scoped to the current site, and does not require a site owner to build anything. Microsoft also says this ready-made agent has no associated .agent file and cannot be edited, deleted, or shared like a custom agent.
  • Custom-built agents are created by editors: A user with edit permissions can create an agent, choose sources, change the agent's name and purpose, and refine how it should answer.
  • Interaction needs licensing, not a separate activation switch: Users need a Microsoft 365 Copilot license, or the organization must enable pay-as-you-go billing for SharePoint agents and scope access through the admin policy.
  • Editing needs both access paths: To edit an agent, a user needs the license or pay-as-you-go path, plus edit permissions on the SharePoint site and the .agent file.
  • Sharing the agent is not the same as sharing the knowledge: If someone can open the agent but cannot access a source document, the agent should not use that restricted document for that person's answer.

That makes the first rollout conversation less about clever prompts and more about SharePoint hygiene: content ownership, source quality, permissions, and sites that should stay away from broad AI discovery until cleaned up.

What Are Agents in SharePoint?

An agent in SharePoint is a scoped AI assistant. It is not a replacement for your intranet, search, or governance model. Think of it as a conversational layer over selected SharePoint content.

  • It answers questions: Users can ask things like "What is the PTO policy?" or "Summarize the project risks in this folder."
  • It uses scoped knowledge: The answer is grounded in the sites, pages, libraries, folders, and files included as sources.
  • It follows user permissions: Answers depend on what the person asking already has permission to access.
  • Custom agents are files: Custom-created agents are stored as .agent files and can be shared like SharePoint files. The ready-made site agent is the exception because Microsoft says it has no associated .agent file.
  • It can live in more than one place: SharePoint-created agents can be used in SharePoint and Teams, and Copilot Studio agents can be deployed to SharePoint as a channel.

What You Need Before You Start

Before you build anything, check these basics. Most failed agent pilots are not technical failures. They fail because licensing, permissions, or content quality was not ready.

  • Microsoft 365 Copilot license or pay-as-you-go enabled: Microsoft documents these as the main access paths for SharePoint agents. Pay-as-you-go access should be scoped through the admin policy and the right security group, not left as an informal rollout decision.
  • Edit permissions to create or modify agents: Visitors can interact where allowed, but site edit permissions are needed to create or edit an agent.
  • Modern SharePoint experience: The creation and use experience is for SharePoint Online modern sites. This is not a SharePoint Subscription Edition/on-premises feature.
  • Clean source content: Agents are only as useful as the pages and documents they can read. Archive stale drafts before using them as sources.
  • Governance decision: Decide who can create agents, who approves them, and which sites are too sensitive for agent discovery.

Important for on-prem SharePoint: SharePoint agents are SharePoint Online/Microsoft 365 capabilities. They do not run inside a SharePoint Subscription Edition farm out of the box. For on-prem environments, the closest pattern is a custom chatbot or Azure Bot/LLM integration that indexes approved content separately.

Option 1 - SharePoint Built-in Agent (No Code)

This is the quickest way to get value. A site owner or member with edit permissions can create an agent in SharePoint, choose sources, give it a purpose, and share it with the team. Start here when the job is mainly answers from SharePoint content, not workflow automation.

Where you can create it

Microsoft lists several entry points for creating a SharePoint agent. Use the one that matches the scope you want. A site-wide agent is good for an intranet or department portal. A document-library agent is better for a project library. A selected-file agent is best when you want tight control over exactly what the agent can use.

  • Site homepage: Choose New > Agent for a site-focused assistant.
  • Document library or list command bar: Use the AI actions menu to create an agent for supported content in that library or list.
  • Selected files or folders: Select a narrow set of files/folders and create an agent from the context menu.
  • Agent chat pane: Open the Copilot button, use the agent dropdown, then create a new agent from there.

What you can customize

  • Name and identity: Give the agent a useful name like "HR Policy Helper" or "Project Phoenix Assistant."
  • Logo/branding: Make it easy for users to recognize which agent they are using.
  • Knowledge sources: Add or remove the sites, pages, libraries, folders, and files the agent should use.
  • Behavior instructions: Tell the agent how to answer, what to avoid, and when to refer users to a human owner.
  • Starter prompts: Give users helpful first questions instead of leaving them staring at an empty chat box.

Example instruction prompt

You do not need a perfect prompt to start. You need a clear purpose and good boundaries.

Agent behavior prompt
You are the HR Policy Helper for Contoso.
Answer only from the HR Policy site, Employee Handbook, Benefits library, and approved HR pages included as sources.
Use simple language and bullet points.
If a policy is unclear, say that the employee should contact HR instead of guessing.
Do not give legal, payroll, or medical advice.
Always cite the source document or page when possible.

Where the .agent file is stored

Custom SharePoint agents are stored as .agent files. That matters because sharing and access to the agent are governed like other SharePoint files.

  • If created from the site homepage, the file is stored under Site contents > Site Assets > Copilots.
  • If created from another location, it is saved in the current document library folder where it was created.
  • Permissions on the .agent file control who can access or edit the agent.
  • The agent still will not reveal source content the user does not have permission to access.

Sharing Agents in Teams and Copilot Chat

Custom-built agents can be shared beyond the SharePoint page. This is useful because a lot of work happens in Teams, not in the document library. A team can ask the project agent for a risk summary during a meeting chat, or a support channel can ask an IT knowledge agent for the current VPN setup instructions.

  • Only custom-built agents can be shared: The ready-made site agent cannot be shared like a custom agent.
  • Use the Teams sharing link: In SharePoint, open the agent list, choose the custom agent, select the ellipsis, and copy the link for Teams.
  • Add it to a chat or channel: After the link is posted in Teams, users are prompted to add the agent to the conversation. Then they can mention the agent in the same style as mentioning a person.
  • Permissions are checked per asker: If the asker can see a source but some people in the Teams chat cannot, Microsoft shows a review flow instead of automatically posting the answer to everyone.
  • Guest scenarios are limited: Microsoft notes that agents refuse to answer if external or guest users are part of the group chat or channel.
  • One-to-one Teams chat is not there yet for SharePoint-created agents: Microsoft's Teams sharing article says users cannot chat 1:1 with agents created from SharePoint yet, and not all Teams mobile features are fully supported.

That review flow is a good thing. Agents make information easier to retrieve, so the product has to be careful about accidentally reposting information into a chat where not everyone has access. Train users to read the response and the cited files before sending an answer to a whole group.

Key Limits to Know

These are the limits and behaviors worth explaining to site owners before they start creating agents.

  • 20 source items: Microsoft Support currently says a SharePoint agent can include up to 20 source items. A source item can be a site, library, folder, or file.
  • Use higher-level sources when needed: If you need more than 20 files, point the agent at a folder or library instead of selecting every file one by one.
  • Supported file types matter: Microsoft lists Office files, PDFs, TXT, RTF, HTML/ASPX, Loop/Fluid, and OpenDocument formats. Images, videos, meeting recordings, and OneNote notebooks are listed as coming soon for SharePoint agent chat scenarios.
  • Pages and Lists are supported scenarios: Microsoft says agents in SharePoint support selecting pages and Lists. For list-heavy use cases, test with real users because column quality, permissions, and the way people phrase questions can change the result.
  • Hub sites can widen the scope: When a SharePoint hub site is included as a source, Microsoft says associated sites are included too. That can be useful, but it also means the agent may cover more content than a site owner expects.
  • Chat history is personal: Microsoft says only the user can see their chat history with agents. Users can rename or delete chat history from the pane.
  • Permissions are still the foundation: If your SharePoint permissions are messy, agents will be messy too.

A simple rule helps: keep the first version narrow. It is better to launch one reliable HR benefits agent over five approved pages than one vague "HR assistant" pointed at every HR folder created in the last ten years.

Option 2 - Custom Copilot Agent in Copilot Studio

Use Copilot Studio when your agent needs to do more than answer questions. This is the right path when the agent must collect details, call Power Automate, connect to external systems, use Dataverse, or follow a controlled topic flow.

This is the advanced path. It can still be low-code, but it usually needs more planning because you are no longer just asking questions over SharePoint content. You are designing an assistant that may authenticate users, call flows, update records, and appear across multiple channels.

Use Copilot Studio when you need

  • Actions: Create a ticket, update a SharePoint list, send an approval, or call a line-of-business API.
  • External data: Use sources beyond SharePoint, such as Dataverse, ServiceNow, Dynamics 365, Salesforce, SQL, or custom connectors.
  • Topic control: Design guided conversations for HR, IT support, procurement, onboarding, or compliance workflows.
  • Enterprise deployment: Test, publish, deploy, approve, monitor, and retire agents through admin-owned processes.
  • Multi-channel availability: Publish to SharePoint, Teams, Microsoft 365 Copilot Chat, Power Pages, or other supported channels.

How Copilot Studio uses SharePoint knowledge

Copilot Studio can add SharePoint as a knowledge source. In Microsoft's current guidance, adding a SharePoint site URL lets the agent search that URL and its subpaths. For example, a site URL can include policy pages and libraries beneath it. Copilot Studio also supports SharePoint lists as knowledge sources, using the user's SharePoint credentials so permission checks still apply.

  • SharePoint site knowledge: Add the SharePoint option from the knowledge dialog, enter one or more URLs, and give each source a useful name and description.
  • SharePoint list knowledge: Add lists through the SharePoint list option. Microsoft says makers can select up to 15 lists at a time, with each list added as an individual knowledge source.
  • Authentication: In Teams, Power Apps, and Microsoft 365 Copilot experiences, Microsoft authentication is normally preconfigured. For manual authentication scenarios, Microsoft documents Entra ID scopes such as Sites.Read.All and Files.Read.All.
  • Filtering: Copilot Studio can filter SharePoint knowledge by fields such as title, author, modified by, and modified date. This is useful for "only show current policy" or "only use files updated this year" patterns.
  • Search restrictions still matter: If your tenant uses Restricted SharePoint Search or Restricted Content Discovery, test the agent with real users because those controls can change what SharePoint content is discoverable for grounding.
  • Grounding discipline: If you want the agent to answer only from the selected SharePoint source, turn off broad web/general-knowledge behavior where appropriate and test what happens when the answer is not found.

For an internal business process, this is where Copilot Studio becomes more powerful than a no-code SharePoint agent. The agent can ask for missing details, confirm them with the user, trigger a Power Automate flow, write to a SharePoint list, and return a confirmation number. That is a different class of solution from content Q&A.

How to deploy a Copilot Studio agent to SharePoint

  1. Build and test the agent in Copilot Studio. Keep SharePoint sources narrow during the first version.
  2. Publish the agent so it is available for channel deployment.
  3. Open Channels and select the SharePoint tile.
  4. Select the SharePoint site or enter the site manually.
  5. Deploy and confirm the deployment.
  6. Test in SharePoint before promoting it to users.
  7. Ask the site owner to approve it if you want it to appear clearly in the site agent picker.

Billing note: Microsoft Learn says a Copilot Studio agent deployed to SharePoint follows Copilot Studio billing policies and consumes Copilot Studio capacity, even when users interact with it from SharePoint.

Licensing difference in plain English

For SharePoint built-in agents, Microsoft documents two access routes for users: a Microsoft 365 Copilot license, or pay-as-you-go billing enabled for SharePoint agents. For Copilot Studio, the maker/admin side is different: organizations generally need a Copilot Studio tenant license and Copilot Studio user licenses for people who create and manage agents. Microsoft also says published agent users do not need a special Copilot Studio maker license just to interact with a published agent, although the organization still has to cover the agent's capacity and channel requirements.

The practical takeaway is simple: do not promise "free agents" just because the button appears in SharePoint. Check whether the users have Microsoft 365 Copilot, whether pay-as-you-go is enabled for unlicensed users, and whether Copilot Studio capacity is in place for advanced agents.

PowerShell for SharePoint Agent Admins

There is not a single magic PowerShell cmdlet that "manages all SharePoint agents" end to end. In practice, admins use three patterns:

  • Inventory agent files because custom SharePoint agents are stored as .agent files.
  • Generate the official Agent Insights report for tenant-wide visibility into where agents are being created.
  • Control risky sites with SharePoint admin controls such as Restricted Content Discovery.

1. Inventory .agent files in a SharePoint site

This PnP PowerShell script scans document libraries in one site and exports discovered .agent files. Run it site by site, or wrap it in a tenant-site loop after testing.

PowerShell 7 - inventory SharePoint .agent files
# Requires PnP.PowerShell
# Install-Module PnP.PowerShell -Scope CurrentUser

$siteUrl = "https://contoso.sharepoint.com/sites/hr"
$outFile = ".\sharepoint-agents-hr.csv"
$clientId = "00000000-0000-0000-0000-000000000000" # Entra app registration for PnP.PowerShell

Connect-PnPOnline -Url $siteUrl -ClientId $clientId -Interactive

$agentFiles = foreach ($library in Get-PnPList | Where-Object { $_.BaseTemplate -eq 101 -and -not $_.Hidden }) {
    Get-PnPListItem -List $library.Title -PageSize 500 -Fields "FileLeafRef", "FileRef", "Editor", "Modified" |
        Where-Object { $_.FieldValues["FileLeafRef"] -like "*.agent" } |
        ForEach-Object {
            [pscustomobject]@{
                SiteUrl  = $siteUrl
                Library  = $library.Title
                Name     = $_.FieldValues["FileLeafRef"]
                Path     = $_.FieldValues["FileRef"]
                Modified = $_.FieldValues["Modified"]
                Editor   = $_.FieldValues["Editor"].Email
            }
        }
}

$agentFiles | Sort-Object Library, Name | Export-Csv -Path $outFile -NoTypeInformation
$agentFiles | Format-Table Library, Name, Modified, Editor -AutoSize

2. Generate an Agent Insights report

Microsoft now documents an Agent Insights report for SharePoint admins. It is available in the SharePoint admin center and SharePoint Online Management Shell, uses Microsoft 365 audit data, and helps admins find sites with recently created agents. Large tenants can take up to 48 hours for data to appear.

SharePoint Online Management Shell - Agent Insights report
Connect-SPOService -Url "https://contoso-admin.sharepoint.com"

# Run these only if your tenant asks you to enable activity data collection first.
Start-SPOAuditDataCollectionForActivityInsights
Get-SPOAuditDataCollectionStatusForActivityInsights

Start-SPOCopilotAgentInsightsReport -ReportPeriodInDays 28
Get-SPOCopilotAgentInsightsReport

$reportId = "00000000-0000-0000-0000-000000000000"
Get-SPOCopilotAgentInsightsReport -ReportId $reportId -Content TopSites
Get-SPOCopilotAgentInsightsReport -ReportId $reportId -Content CopilotAgentsOnSites -Action Download

3. Turn off agent-related features on a sensitive site

Microsoft documents Restricted Content Discovery as a site-level control for high-risk SharePoint sites. It can keep site content out of organization-wide search and Microsoft 365 Copilot discovery, and Microsoft says users will not see the Copilot icon on that site. It is not a permissions cleanup tool, does not apply to OneDrive, and can take time to show across experiences.

SharePoint Online Management Shell - restrict a sensitive site
# Requires the latest Microsoft SharePoint Online Management Shell
$tenant = "contoso"
$siteUrl = "https://contoso.sharepoint.com/sites/executive"

Connect-SPOService -Url "https://$tenant-admin.sharepoint.com"

# Optional: allow delegated management of Restricted Content Discovery.
Set-SPOTenant -DelegateRestrictedContentDiscoverabilityManagement $true
Get-SPOTenant | Select-Object DelegateRestrictedContentDiscoverabilityManagement

# Enable Restricted Content Discovery for a high-risk site
Set-SPOSite -Identity $siteUrl -RestrictContentOrgWideSearch $true

# Confirm status
Get-SPOSite -Identity $siteUrl | Select-Object Url, RestrictContentOrgWideSearch

# Later, after permissions and content are cleaned up, remove the restriction.
# Set-SPOSite -Identity $siteUrl -RestrictContentOrgWideSearch $false

Use this setting selectively. Microsoft's Restricted Content Discovery documentation says it can protect high-risk sites from tenant-wide search and Copilot discovery, but overuse can reduce the quality of search and Copilot answers because there is less content available for grounding.

4. Generate a Restricted Content Discovery report

Use this when you want evidence of which sites have the control enabled.

PowerShell - report on Restricted Content Discovery
Connect-SPOService -Url "https://contoso-admin.sharepoint.com"

# Start the report
Start-SPORestrictedContentDiscoverabilityReport

# View report jobs
Get-SPORestrictedContentDiscoverabilityReport

# Download a completed report by ReportId
Get-SPORestrictedContentDiscoverabilityReport -Action Download -ReportId "00000000-0000-0000-0000-000000000000"

Governance Checklist Before Rollout

Do this before you encourage everyone to create agents. It is much easier to set rules early than to clean up hundreds of unmanaged agents later.

  • Name ownership clearly: Every approved agent should have a named business owner and technical owner.
  • Define source rules: Decide which sites are allowed as knowledge sources and which are blocked or restricted.
  • Review permissions first: Fix broad access, stale guests, "Everyone except external users", and anonymous sharing links before adding sources.
  • Use approval for important agents: Site owners can approve agents so visitors can distinguish official agents from ad hoc ones.
  • Set a site default only when it helps: Site owners can set an approved agent as the default agent for the site. Use this for official assistants, not experiments.
  • Watch the .agent files: Custom agents are files. Include them in content lifecycle, retention, ownership, and permission reviews.
  • Use Agent Insights: Review the SharePoint admin center Agent Insights report or the PowerShell report regularly so unmanaged agent growth does not surprise you.
  • Monitor billing: For pay-as-you-go, create Azure budgets and cost alerts before broad rollout, especially if unlicensed users will use SharePoint agents.
  • Keep prompts plain: The best instructions are specific, boring, and testable.
  • Set a review date: Recheck agent sources and permissions every quarter, or whenever a project closes.
  • Teach users to verify: AI-generated answers can be incomplete or outdated. Users should open citations for decisions that matter.

Example Use Cases

  • HR policy helper: Answers questions about leave, benefits, onboarding, and performance review processes from approved HR pages and files.
  • Project knowledge assistant: Summarizes status reports, decision logs, risks, and meeting notes from one project library.
  • IT self-service agent: Handles password reset instructions, VPN setup, approved software requests, and known issue lookups.
  • Procurement assistant: Explains approval thresholds, preferred supplier rules, and contract template guidance.
  • Compliance playbook agent: Helps employees find the right policy, checklist, or control evidence without searching multiple sites.

Common Mistakes to Avoid

  • Pointing agents at everything: Wider scope is not always better. Narrow, curated sources usually produce clearer answers.
  • Skipping permissions cleanup: Agents respect permissions, but they also make existing oversharing much easier to discover.
  • Using vague prompts: "Help users with HR" is weak. "Answer only from these HR policy sources, cite documents, and refer unclear cases to HR" is useful.
  • Confusing built-in agents with Copilot Studio: Built-in SharePoint agents are great for content Q&A. Copilot Studio is better for workflow and integration.
  • Forgetting billing: If you use pay-as-you-go, Copilot Studio capacity, or advanced agents, monitor usage before broad rollout.
