Automate SharePoint Site Provisioning End to End

User request → manager approval → Logic App orchestration → PnP PowerShell runbook → error logging → instant email notification. The complete pipeline.

Admin & Automation June 2, 2026 14 min read

Every SharePoint admin knows the pain: someone requests a new site, a ticket is raised, IT gets to it three days later, the site is created manually without a standard structure, and the requester gets a URL in a chat message. This guide replaces that entire process with a self-service pipeline that provisions a properly structured SharePoint site in under five minutes — with manager approval, audit logging, error recovery, and email notifications built in from the start.

What We're Building

The pipeline is made up of six components working in sequence. Each has a single, clear responsibility, which makes the whole system easy to maintain and easy to debug when something goes wrong.

📋
1 — User submits a request

A SharePoint list form (or embedded Power Apps canvas app) captures site name, URL alias, type, template, owner, and business purpose. The item is created with Status = Pending Approval.

✅
2 — Power Automate routes to manager

A cloud flow detects the new item, looks up the requester's line manager from Azure AD, and sends an Adaptive Card approval to the manager's Teams and email. A 48-hour timeout triggers a reminder and notifies the requester.

⚡
3 — Azure Logic App orchestrates creation

On approval, Power Automate POSTs the request to a Logic App HTTP trigger. The Logic App validates the URL, calls an Azure Automation runbook, polls for completion, and handles all success and failure branches.

🔧
4 — Azure Automation runs the PnP PowerShell script

The runbook connects via Managed Identity, creates the site, adds owners, applies the correct template structure (document libraries, lists, folders), and sets the sharing policy. It returns the provisioned URL or throws a descriptive error.

📝
5 — Errors are logged to SharePoint

If any step fails, the Logic App catch scope writes a structured entry to a Site Provision Logs list — including which step failed, the error message, and the Azure Automation job ID for deep-dive tracing.

📧
6 — Requester gets an email regardless of outcome

Success delivers the site URL and quick-start tips. Rejection delivers the manager's reason. Failure delivers a reference ID and a support link — no one is left waiting and wondering.

Step 1: Create the SharePoint Lists

Two SharePoint lists underpin the entire pipeline. Create them in a central IT or admin site collection — not in a personal or team site.

Site Requests list

This is the intake form and the live status board. Every provisioning request flows through it from submission to completion.

Step 2: Manager Approval Flow (Power Automate)

Create a new Automated cloud flow in Power Automate named SP Site Request — Manager Approval. This flow runs entirely within your Microsoft 365 tenant — no Azure subscription required.

Trigger and guard condition

Trigger: When an item is created (SharePoint connector) — pointing to the Site Requests list.

Immediately after the trigger, add a Condition that checks Status Value equals Pending Approval. This guards against re-triggering if an item is modified after the flow runs.

Look up the manager

Add a Get manager (V2) action from the Office 365 Users connector. Pass the requester's email from the trigger. This resolves the correct approver automatically — the flow never needs a hardcoded email.

Send the approval

Add a Start and wait for an approval action. Configure it as follows:

FieldValue
Approval typeApprove/Reject — First to respond
TitleSite request: @{triggerOutputs()?['body/Title']}
Assigned to@{outputs('Get_manager_(V2)')?['body/mail']}
DetailsRequester: [Created By name]
URL alias: [SiteURLAlias]
Type: [SiteType]
Purpose: [BusinessPurpose]
Item linkLink to the SharePoint list item
TimeoutP2D (ISO 8601 for 48 hours)

Branch: Approved

In the Yes branch of a Condition on Outcome equals Approve:

  1. Update the Site Requests item: Status = Approved, ManagerNotes = approval comment
  2. Add an HTTP action (POST) to the Logic App HTTP trigger URL, passing a JSON body with all site request fields. This fires the Logic App without the flow waiting for it to complete.

Branch: Rejected

In the No branch:

  1. Update the item: Status = Rejected, ManagerNotes = rejection comment
  2. Send an email to the requester:

Rejection email template

Subject: Your SharePoint site request was not approved

Hi [Requester first name],

Your request for a SharePoint site named "[Site Title]" was reviewed by [Manager name] and was not approved at this time.

Reason: [ManagerNotes]

If you'd like to resubmit with amendments, click the link to the Site Requests list in your confirmation email. If you have questions, reply to this email.

Branch: Timed out

Use Configure run after on a parallel branch set to run when the approval action has timed out. In this branch: send a reminder email to the manager and a holding email to the requester ("We've chased your manager and are waiting for a response").

Step 3: Azure Logic App — The Orchestrator

The Logic App is the heart of the pipeline. It owns every step after approval: URL validation, runbook invocation, polling, error capture, and all notifications. Create a Consumption plan Logic App named sp-site-provisioning-orchestrator.

HTTP trigger

Add a When a HTTP request is received trigger. Use Use sample payload to generate schema and paste a representative JSON body to get the schema auto-generated. Copy the generated URL and paste it into the Power Automate HTTP action from Step 2.

Sample JSON body from Power Automate
{
  "requestItemId": 42,
  "siteTitle": "Marketing Hub 2026",
  "siteUrlAlias": "marketing-hub-2026",
  "siteType": "CommunicationSite",
  "template": "General",
  "primaryOwnerEmail": "jane.smith@contoso.com",
  "secondaryOwnerEmail": "john.doe@contoso.com",
  "description": "Central hub for the Marketing department",
  "requesterEmail": "jane.smith@contoso.com",
  "tenantName": "contoso"
}

URL availability check

Before invoking the runbook, verify the URL alias is not already taken. Add an HTTP action calling the SharePoint REST API:

HTTP GET — check site existence
GET https://@{body('Parse_JSON')?['tenantName']}-admin.sharepoint.com
    /_api/SPO.Tenant/GetSitePropertiesByUrl(
      url='https://@{body('Parse_JSON')?['tenantName']}.sharepoint.com
           /sites/@{body('Parse_JSON')?['siteUrlAlias']}',
      includeDetail=false)

Authentication: Managed Identity
Audience: https://@{body('Parse_JSON')?['tenantName']}-admin.sharepoint.com

Follow this with a Condition: if the HTTP response Status code equals 200, the site already exists — route to an error log entry and a "URL taken" email to the requester, then terminate. Any other status code means the URL is free.

Try / Catch pattern with Scope actions

Wrap all remaining steps inside a Scope action named Try. Then add a second Scope named Catch configured via Configure run after to run only when Try has failed. This is the Logic App equivalent of try/catch and ensures errors are always handled.

Inside the Try scope:

  1. Update item (SharePoint) — set Status = Provisioning
  2. Create job (Azure Automation) — invoke the SP-SiteProvisioning runbook, passing all site parameters. Store the returned Job ID in a variable.
  3. Update item — write the Job ID to LogicAppRunID
  4. Do Until loop — poll the job every 30 seconds (Delay action), using Get job to check its status. Exit when status equals Completed, Failed, or Suspended, or when the loop counter exceeds 20 (10 minutes maximum).
  5. After the loop, check the final job status with a Condition:
    • Completed: Update item (Status = Provisioned, ProvisionedURL = full site URL) → send success email
    • Anything else: use Terminate with status Failed and a descriptive message — this causes the Try scope to fail, triggering the Catch scope

Inside the Catch scope (runs only on Try failure):

  1. Extract the error using the result('Try') expression
  2. Create a row in the Site Provision Logs list
  3. Update the Site Requests item (Status = Failed)
  4. Send the failure email to the requester
Logic App expression — extract error from Try scope
# Error message
@{result('Try')?[0]?['error']?['message']}

# Failed action name
@{result('Try')?[0]?['name']}

# Full error code
@{result('Try')?[0]?['error']?['code']}

Step 4: Azure Automation Runbook (PnP PowerShell)

The runbook is a PowerShell 7.2 runbook in an Azure Automation account. It does all the actual SharePoint work: creates the site, adds owners, applies the template structure, and locks down sharing settings.

Prerequisites

  • Azure Automation account with PnP.PowerShell imported from the module gallery (search for PnP.PowerShell under Modules > Browse Gallery)
  • System-assigned Managed Identity enabled on the Automation account
  • The Managed Identity's Object ID granted SharePoint Administrator role in Microsoft 365 admin centre, or granted application permission Sites.FullControl.All on Microsoft Graph via Entra app consent

Never store credentials in a runbook. Use Managed Identity exclusively. If your Automation account cannot use Managed Identity (older accounts), use a Certificate credential stored in Azure Key Vault, fetched at runtime via Get-AutomationCertificate.

Runbook: SP-SiteProvisioning

PowerShell 7.2 runbook — SP-SiteProvisioning
param(
    [Parameter(Mandatory=$true)] [string]$TenantName,
    [Parameter(Mandatory=$true)] [string]$SiteTitle,
    [Parameter(Mandatory=$true)] [string]$SiteUrlAlias,
    [Parameter(Mandatory=$true)] [string]$SiteType,
    [Parameter(Mandatory=$true)] [string]$Template,
    [Parameter(Mandatory=$true)] [string]$PrimaryOwnerEmail,
    [string]$SecondaryOwnerEmail = "",
    [string]$Description = "",
    [Parameter(Mandatory=$true)] [string]$RequestItemId
)

$ErrorActionPreference = 'Stop'

$adminUrl   = "https://$TenantName-admin.sharepoint.com"
$siteFullUrl = "https://$TenantName.sharepoint.com/sites/$SiteUrlAlias"

Write-Output "INFO: Starting provisioning for $siteFullUrl"

try {
    # --- Connect to SharePoint admin via Managed Identity ---
    Connect-PnPOnline -Url $adminUrl -ManagedIdentity
    Write-Output "INFO: Connected to admin centre"

    # --- Create the site ---
    if ($SiteType -eq "CommunicationSite") {
        New-PnPSite -Type CommunicationSite `
            -Title       $SiteTitle `
            -Url         $siteFullUrl `
            -Owner       $PrimaryOwnerEmail `
            -Description $Description `
            -SiteDesign  Blank `
            -Wait
    } else {
        New-PnPSite -Type TeamSiteWithoutMicrosoft365Group `
            -Title       $SiteTitle `
            -Url         $siteFullUrl `
            -Owner       $PrimaryOwnerEmail `
            -Description $Description `
            -Wait
    }
    Write-Output "INFO: Site created at $siteFullUrl"

    # --- Connect to the new site ---
    Connect-PnPOnline -Url $siteFullUrl -ManagedIdentity

    # --- Add secondary owner ---
    if ($SecondaryOwnerEmail) {
        Add-PnPSiteCollectionAdmin -Owners @($SecondaryOwnerEmail)
        Write-Output "INFO: Added secondary owner $SecondaryOwnerEmail"
    }

    # --- Apply template structure ---
    switch ($Template) {
        "Project Team" {
            New-PnPList -Title "Project Tasks" -Template GenericList -ErrorAction SilentlyContinue
            New-PnPList -Title "Risk Register"  -Template GenericList -ErrorAction SilentlyContinue
            Add-PnPFolder -Name "Project Plans"   -Folder "Shared Documents"
            Add-PnPFolder -Name "Deliverables"    -Folder "Shared Documents"
            Add-PnPFolder -Name "Meeting Notes"   -Folder "Shared Documents"
            Add-PnPFolder -Name "Archive"         -Folder "Shared Documents"
        }
        "HR Department" {
            New-PnPList -Title "Job Postings" -Template GenericList -ErrorAction SilentlyContinue
            Add-PnPFolder -Name "Policies"    -Folder "Shared Documents"
            Add-PnPFolder -Name "Templates"   -Folder "Shared Documents"
            Add-PnPFolder -Name "Archive"     -Folder "Shared Documents"
            # HR sites never share externally
            Set-PnPTenantSite -Url $siteFullUrl -SharingCapability Disabled
        }
        "IT Service Desk" {
            New-PnPList -Title "Tickets"      -Template GenericList -ErrorAction SilentlyContinue
            New-PnPList -Title "Known Issues" -Template GenericList -ErrorAction SilentlyContinue
            Add-PnPFolder -Name "Runbooks"    -Folder "Shared Documents"
            Add-PnPFolder -Name "Procedures"  -Folder "Shared Documents"
        }
        default {
            # General — minimal structure
            Add-PnPFolder -Name "Working Documents" -Folder "Shared Documents"
            Add-PnPFolder -Name "Archive"           -Folder "Shared Documents"
        }
    }
    Write-Output "INFO: Template '$Template' applied"

    # --- Default sharing policy (override per template above) ---
    if ($Template -ne "HR Department") {
        Set-PnPTenantSite -Url $siteFullUrl `
            -SharingCapability ExistingExternalUserSharingOnly
    }

    Write-Output "PROVISIONED:$siteFullUrl"

} catch {
    $msg = $_.Exception.Message
    Write-Error "FAILED at RequestID $RequestItemId — $msg"
    throw "Provisioning failed for $siteFullUrl — $msg"
}

A few design decisions worth noting:

  • $ErrorActionPreference = 'Stop' — converts all non-terminating errors into terminating ones, so every failure is caught by the catch block and propagated back to the Logic App as a job failure.
  • -Wait on New-PnPSite — blocks the runbook until the site is fully provisioned before executing subsequent steps. Without it, Connect-PnPOnline to the new site will fail because the site isn't ready yet.
  • TeamSiteWithoutMicrosoft365Group — creates a SharePoint-only team site with no linked Microsoft 365 Group or Teams team. Use this unless the requester explicitly needs a Group-connected workspace.
  • Write-Output "PROVISIONED:..." — the Logic App's Get job output action reads the runbook's stdout. Prefixing the final output with a known token makes it trivial to extract the URL via a Logic App expression.

Step 5: Error Logging to SharePoint

Errors are inevitable — a site URL with a reserved word triggers a 409, a transient Microsoft API hiccup returns a 503, the PnP module has a version mismatch. The pipeline is designed so that none of these errors silently disappear.

The Logic App's Catch scope creates a row in the Site Provision Logs list. The key expressions used in that action are:

Logic App — Catch scope expressions for log entry
# Title (unique identifier for the log row)
ERR-@{variables('requestItemId')}-@{formatDateTime(utcNow(),'yyyyMMdd-HHmmss')}

# Which action inside Try scope failed
@{result('Try')?[0]?['name']}

# Human-readable error message
@{result('Try')?[0]?['error']?['message']}

# Full error code / raw detail
@{result('Try')?[0]?['error']?['code']}

# Azure Automation job ID (stored in variable during Try scope)
@{variables('runbookJobId')}

With these entries in the log list, an admin can:

  • Filter by RequestID to see the full history for one request
  • Filter by LogLevel = Error and sort by date to see all recent failures
  • Copy the RunbookJobID and paste it into Azure Automation > Jobs to see the full PowerShell output and stack trace
  • Set a Power Automate alert on new rows with LogLevel = Error to notify the IT team immediately

Retention tip: Apply a SharePoint retention label to the Provision Logs list that deletes rows older than 12 months. This keeps the list fast to query and stops it growing unbounded over time.

Step 6: End-User Notifications

Every outcome sends the requester an email. No one is left guessing whether their request was processed.

Success notification (sent from Logic App Try scope)

Subject: Your SharePoint site is ready — [Site Title]

Hi [First Name],

Your new SharePoint site has been created and is ready to use.

Site name: [SiteTitle]
URL: [ProvisionedURL — inserted dynamically by Logic App]
Template applied: [Template]
Primary owner: [PrimaryOwnerEmail]

Getting started: Sign in, invite your team members via site Settings > Site permissions, and review the default document library structure that's been set up for you.

Questions? Contact the IT helpdesk.

Rejection notification (sent from Power Automate)

Subject: Site request update — [Site Title] was not approved

Hi [First Name],

Your request for a SharePoint site named "[Site Title]" was reviewed by your manager and was not approved at this time.

Manager's comment: [ManagerNotes]

You can amend and resubmit your request via the Site Requests list. If you believe this decision was made in error, please speak to your manager directly.

Failure notification (sent from Logic App Catch scope)

Subject: Site creation issue — reference [RequestItemId]

Hi [First Name],

We ran into a technical problem while creating your SharePoint site "[Site Title]". Our team has been notified automatically.

Reference number: [RequestItemId]
Failed step: [FailedStep from log entry]

You do not need to resubmit your request — the IT team will investigate and either resolve the issue or contact you within one business day. If you need to escalate, quote reference [RequestItemId] to the helpdesk.

Testing the Pipeline End to End

Before going live, run each scenario deliberately. Don't rely on happy-path testing alone — the error paths are the most important to verify.

Test scenarioHow to trigger itExpected outcome
Happy pathSubmit a valid request with a fresh URL alias and approve itSite provisioned, success email received, Status = Provisioned, ProvisionedURL populated
RejectionSubmit a request and reject it in the approval card with a commentStatus = Rejected, ManagerNotes populated, rejection email with reason delivered
URL conflictSubmit a request using the alias of an existing siteLogic App exits early, error log entry created, "URL taken" email sent to requester
Runbook failureTemporarily revoke the Managed Identity's SharePoint permission, then submit and approve a requestCatch scope fires, error logged to Provision Logs with the auth error, failure email sent
Approval timeoutReduce the approval timeout to PT2M for testing, do not respondReminder email to manager, holding email to requester delivered after timeout

After each test, verify three things: the SharePoint item Status column updated correctly, the correct email was received by the requester, and (for failure cases) a row appeared in the Provision Logs list with the right content.

Governance and Maintenance Tips

A provisioning pipeline creates sites quickly — which means it also creates governance debt quickly if left unchecked.

URL alias validation in Power Automate

Before even routing for approval, add a Condition in the Power Automate flow that validates the URL alias against a regex: ^[a-z0-9][a-z0-9\-]{0,38}[a-z0-9]$. If it fails, immediately set Status = Failed and send the requester a validation error email, saving the manager from approving something that will always fail at the runbook stage.

Reserved URL prefix enforcement

Maintain a small SharePoint list of reserved prefixes (e.g. hr-, legal-, exec-) that require IT sign-off before provisioning. The Logic App checks the alias against this list before invoking the runbook, and if matched, flags the item for manual review.

Site lifecycle and expiry

Every provisioned site should have an expected lifetime. Add an ExpectedExpiryDate date column to the Site Requests list and populate it at provisioning time based on template type (Project = 18 months, HR = no expiry, General = 12 months). A separate scheduled Power Automate flow queries for sites expiring within 30 days and emails the primary owner to either extend or confirm deletion.

Add a Teams team toggle

If a requester needs Microsoft Teams alongside the SharePoint site, change the runbook's New-PnPSite call to use -Type TeamSite (which creates an M365 Group-connected site) and add a New-Team call via the Microsoft Teams PowerShell module. Surface this as a "Create linked Teams team?" Yes/No toggle in the request form — the Logic App passes it as a parameter to the runbook.
