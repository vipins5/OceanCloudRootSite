InfoPath Forms Services Deadline: Render XML Forms to PDF

Before July 14, 2026, render every historical InfoPath XML form to PDF and store those PDFs in a governed SharePoint document library.

Admin & Automation July 1, 2026 14 min read

InfoPath is not just another retired Office client. In many SharePoint tenants, InfoPath still controls how old HR requests, finance approvals, safety checks, legal intakes, and regulated business records are displayed. The urgent work is not only migration. It is preservation: create a readable PDF copy of every historical XML form while the browser rendering service is still available.

Deadline: Treat July 14, 2026 as the last safe date for browser-rendering historical InfoPath XML forms in SharePoint Online. Finish PDF rendering, metadata capture, and validation before that date.

Official Microsoft Details to Know

Microsoft has been clear that InfoPath is an older technology and that organizations should move to modern alternatives. The public retirement story has three parts:

  • InfoPath Forms Services in Microsoft 365 is retiring. Microsoft announced through the SharePoint Blog that InfoPath Forms Services support in Microsoft 365 would continue only until the Office 2016 support date of July 14, 2026.
  • InfoPath 2013 reaches end of support on July 14, 2026. The Microsoft Lifecycle page lists InfoPath 2013 support ending on that date.
  • Message Center updates add operational deadlines. Tenant admins should review the official Microsoft 365 admin center Message Center posts for their tenant, including InfoPath Forms Services retirement updates. Recent Message Center communications describe a May 18, 2026 publishing block for new or updated InfoPath forms and full retirement on July 14, 2026.

The practical interpretation is simple: if a submitted XML file must remain readable for audit, legal, finance, HR, or customer service reasons, render it to PDF before the service disappears.

Do not rely on XML alone. XML is the data. The XSN template, rules, views, formatting, repeating sections, and attachment handling are what make the form understandable to a human reader.

What You Need to Preserve

A good archive does not throw away the original source. It keeps the readable copy and the technical source together.

ItemWhy it mattersWhere to store it
Rendered PDFThe durable, human-readable record for business users, auditors, and records managers.New SharePoint document library, usually one per site or business process.
Original XMLThe submitted data file and source evidence.Keep in the original library or copy to a protected archive location.
XSN templateThe view and rules that make the XML understandable.Archive library folder named by form template and version.
AttachmentsInfoPath attachments may be stored inside XML or referenced by the form process.Extract or preserve according to your records policy.
Manifest CSVMaps each PDF back to source site, library, XML file, template, submitter, date, hash, and conversion status.Same records library, restricted to owners and records admins.

Step 1: Inventory Every InfoPath Form Library

Start with discovery. You need to know which sites have form libraries, which libraries contain XML submissions, and which templates are still active.

  1. 1
    Run the Microsoft 365 Assessment tool

    Use Microsoft's assessment tooling to find InfoPath usage at tenant scale. Treat its output as the starting inventory, then confirm high-risk sites manually.

  2. 2
    Ask business owners to confirm retention need

    Some libraries are abandoned and can follow normal retention cleanup. Others are legal records and must be preserved.

  3. 3
    Classify forms by risk

    Prioritize regulated records, approval histories, employee files, finance submissions, customer commitments, and forms with attachments.

PowerShell starter inventory

This PnP PowerShell snippet gives admins a lightweight way to find likely InfoPath form libraries. Run it in a pilot first, then expand across the tenant.

PowerShell 7 - find form libraries
$adminUrl = "https://contoso-admin.sharepoint.com"
$clientId = "00000000-0000-0000-0000-000000000000"

Connect-PnPOnline -Url $adminUrl -Interactive -ClientId $clientId

$sites = Get-PnPTenantSite -Limit All |
  Where-Object { $_.Template -notlike "SPSPERS*" }

$inventory = foreach ($site in $sites) {
  Write-Host "Checking $($site.Url)" -ForegroundColor Cyan
  Connect-PnPOnline -Url $site.Url -Interactive -ClientId $clientId

  Get-PnPList -Includes RootFolder,ItemCount,BaseTemplate,Hidden |
    Where-Object {
      -not $_.Hidden -and (
        $_.BaseTemplate -eq 115 -or
        $_.RootFolder.ServerRelativeUrl -match "/Forms$" -or
        $_.Title -match "form|infopath"
      )
    } |
    ForEach-Object {
      [pscustomobject]@{
        SiteUrl   = $site.Url
        ListTitle = $_.Title
        ItemCount = $_.ItemCount
        ListUrl   = $_.RootFolder.ServerRelativeUrl
        Template  = $_.BaseTemplate
      }
    }
}

$inventory |
  Sort-Object SiteUrl, ListTitle |
  Export-Csv .\infopath-form-library-inventory.csv -NoTypeInformation

Step 2: Create the PDF Archive Library

Create a dedicated library before conversion begins. Do not drop PDFs back into the old form library without structure. You want a clean records destination with columns that make search, audit, and validation easy.

ColumnTypePurpose
OriginalFormUrlHyperlink or single line textPoints to the source XML record.
OriginalFileNameSingle line textPreserves the old XML file name.
FormTemplateSingle line textNames the XSN or business process.
SubmittedByPerson or single line textCaptures the original submitter when available.
SubmittedDateDate and timeSupports retention and records lookup.
RenderedOnDate and timeShows when the PDF was created.
SourceXmlHashSingle line textProvides evidence that the archive points to a specific source file.
ConversionStatusChoiceValues such as Rendered, Failed, Needs Review, Not Required.
PowerShell 7 - create archive library columns
$siteUrl = "https://contoso.sharepoint.com/sites/records"
$libraryTitle = "InfoPath PDF Archive"
$clientId = "00000000-0000-0000-0000-000000000000"

Connect-PnPOnline -Url $siteUrl -Interactive -ClientId $clientId

New-PnPList -Title $libraryTitle -Template DocumentLibrary -OnQuickLaunch:$true

Add-PnPField -List $libraryTitle -DisplayName "OriginalFormUrl" -InternalName "OriginalFormUrl" -Type URL
Add-PnPField -List $libraryTitle -DisplayName "OriginalFileName" -InternalName "OriginalFileName" -Type Text
Add-PnPField -List $libraryTitle -DisplayName "FormTemplate" -InternalName "FormTemplate" -Type Text
Add-PnPField -List $libraryTitle -DisplayName "SubmittedBy" -InternalName "SubmittedBy" -Type Text
Add-PnPField -List $libraryTitle -DisplayName "SubmittedDate" -InternalName "SubmittedDate" -Type DateTime
Add-PnPField -List $libraryTitle -DisplayName "RenderedOn" -InternalName "RenderedOn" -Type DateTime
Add-PnPField -List $libraryTitle -DisplayName "SourceXmlHash" -InternalName "SourceXmlHash" -Type Text
Add-PnPField -List $libraryTitle -DisplayName "ConversionStatus" -InternalName "ConversionStatus" -Type Choice -Choices "Rendered","Failed","Needs Review","Not Required"

Step 3: Freeze InfoPath Publishing

By this stage of retirement, the safest operating model is a freeze. Stop form template edits, stop new InfoPath development, and route change requests to the replacement program.

  • Freeze XSN changes: no design updates unless required to recover records before the deadline.
  • Export every template: store XSN files with template name, site, library, version, owner, and archive date.
  • Stop new intake: redirect active business processes to the replacement form or a temporary controlled intake method.
  • Capture ownership: each form library needs a business owner who signs off on archive completeness.

Step 4: Render Every Historical XML Form to PDF

This is the main work. Open each submitted XML form using the same view users would normally see, render it to PDF, and upload the PDF to the archive library with metadata.

There is no universal one-click Microsoft conversion that turns every InfoPath XML file into a perfect PDF across every custom template. Some tenants can automate parts of the job with browser automation, print-to-PDF, or approved third-party records tools. Others need a supervised batch process because old templates include code, custom views, attachments, or data connections.

Recommended control: convert in batches by library, validate each batch count, and keep failed XML records in a Needs Review queue until a human confirms the reason.

Conversion checklist

  • Open the form as a user with permission to see all required data.
  • Select the correct view if the template has multiple views.
  • Check repeating sections, totals, conditional sections, and attachment areas.
  • Export or print the rendered form to PDF with a stable file name.
  • Upload the PDF to the archive library.
  • Write source URL, original XML file name, rendered date, template name, submitter, and conversion status.
  • Record failures and retest before July 14, 2026.

Step 5: Validate Counts, Metadata, and Search

Do not declare success because "the script finished." Records migration needs evidence.

ValidationPass condition
Count matchEvery in-scope XML file has a PDF, or a documented Not Required / Failed status.
Metadata matchPDF library columns map back to source URL, original file name, submitted date, and template.
Open testSample PDFs open in browser and download correctly.
Search testRecords team can find PDFs by business key, submitter, date, and process name.
Retention testLabels and library permissions align with your Microsoft Purview or records policy.

Step 6: Replace Active InfoPath Forms

Archiving solves historical readability. It does not replace active business processes. Use the simplest modern tool that fits the process:

  • Power Apps: best for SharePoint list forms, conditional screens, richer validation, role-based behavior, and mobile-friendly business apps.
  • Microsoft Forms: best for simple intake, surveys, questionnaires, and lightweight collection where the form does not need SharePoint-style app logic.
  • Power Automate: best for routing, approvals, notifications, reminders, integration, and audit logging.
  • SharePoint lists and libraries: still the right storage layer for many structured business processes after the form front end is modernized.

Migration Priority Matrix

If time is short, start with forms that are both high-risk and hard to reconstruct after retirement.

PriorityExamplesAction before July 14, 2026
